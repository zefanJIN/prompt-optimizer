# 快速开始

这份快速开始只做一件事：帮你用最短路径跑通一次真实工作流。

!!! warning
    Prompt Optimizer 不会替你内置可直接使用的模型。第一次使用前，至少要先在右上角 **模型管理** 里配置 1 个可用的文本模型，否则左侧分析 / 优化和右侧测试 / 评估都跑不起来。

## 第一次使用，只做这 4 步

1. 先在右上角 **模型管理** 配 1 个文本模型
2. 进入适合你的工作区
3. 左侧先做一次 **优化** 或 **分析**
4. 右侧再跑 **测试 / 结果评估 / 对比评估**

如果你现在最不确定的是：

- 模型怎么配：看 [模型管理](../basic/models.md)
- 应该进哪个工作区：看 [选择工作区](choose-workspace.md)
- “分析”和“评估”有什么区别：看 [测试与评估](testing-evaluation.md)

## 第一步：先配一套最小可用模型

- 文本工作区：至少 1 个文本模型
- 图像工作区：至少 1 个文本模型 + 1 个图像模型
- 多图生图：还要确认右侧图像模型支持多图输入，并准备至少两张图片

第一次使用，最推荐的做法很简单：先只配 1 个文本模型，把下面这些动作跑通：

1. 左侧分析 / 优化
2. 右侧测试
3. 右侧结果评估
4. 右侧对比评估

确认流程没问题后，再补第 2 个文本模型做对照测试。

更完整的说明见 [模型管理](../basic/models.md) 和 [模型选择与测试策略](model-testing-strategy.md)。

## 第二步：选对工作区

| 工作区 | 你在优化什么 | 右侧主要输入 |
| --- | --- | --- |
| [系统提示词工作区](../basic/system-optimization.md) | 角色、规则、边界、输出规范 | 一段测试文本 |
| [用户提示词工作区](../basic/user-optimization.md) | 一条直接发给模型的任务提示词 | 通常不需要额外输入 |
| [变量工作区](../advanced/variables.md) | 一条可复用的变量化模板 | 一组变量值 |
| [多消息工作区](../advanced/context.md) | 完整会话里的某一条 `system/user` 消息 | 整段会话 + 共享变量 + 可选工具 |
| [文生图工作区](../image/text2image-workspace.md) | 图像提示词 | 图像模型 |
| [图生图工作区](../image/image2image-workspace.md) | 带输入图的图像提示词 | 输入图 + 图像模型 |
| [多图生图工作区](../image/multiimage-workspace.md) | 多图关系提示词 | 至少两张输入图 + 图像模型 |

## 第三步：理解左侧和右侧各管什么

这是 Prompt Optimizer 最重要的一层语义：

- **左侧**：改提示词
- **右侧**：跑真实结果
- **右侧评估**：基于真实执行证据判断结果好不好

| 区域 | 主要动作 | 目标 |
| --- | --- | --- |
| 左侧原始区 | 输入原始提示词或原始消息 | 给优化一个起点 |
| 左侧工作区 | 承接优化、分析、迭代后的当前可编辑内容 | 继续修改并形成版本链 |
| 右侧测试区 | 执行 `原始 / 工作区 / vN` | 看真实输出 |
| 右侧结果评估 | 看单列结果是否合格 | 定位单次执行问题 |
| 右侧对比评估 | 看多列差异 | 总结更好的模式 |

更完整的边界说明见 [测试与评估](testing-evaluation.md)。

## 第四步：跑一次最小工作流

无论哪个文本工作区，都可以先按这条顺序走：

1. 左侧输入原始内容
2. 左侧先点 **优化** 或 **分析**
3. 让结果进入下方工作区
4. 右侧选择要测试的版本或工作区
5. 点击 **测试全部** 或单列运行
6. 先做 **结果评估**，再做 **对比评估**
7. 把真正有价值的建议应用回左侧工作区

## 如果连不上模型，再看部署与连接问题

最简单的判断表：

| 你的情况 | 推荐方式 | 为什么 |
| --- | --- | --- |
| 只想快速体验，主要连公开 HTTPS API | 在线站 / Web 版 | 最省事 |
| 要连 Ollama、LM Studio、局域网或内网接口 | 桌面版 | 不受浏览器跨域和 Mixed Content 限制 |
| 要自托管 Web 页面，并顺带暴露 MCP | Docker | 一次起 Web + `/mcp` |

对应文档：

- [Web 版部署](../deployment/web.md)
- [桌面应用](../deployment/desktop.md)
- [Docker 基础部署](../deployment/docker-basic.md)

!!! warning
    当前 Web 版不是“内置代理服务”。无论在线站还是自部署静态站，浏览器都会直接向你配置的模型服务发请求。

## 第六步：用三个最小例子跑通

### 例子 A：系统提示词工作区

进入 `/#/basic/system`。

左侧输入：

```text
你是一个技术支持助手，请回答用户问题。
```

右侧测试文本输入：

```text
我在 Windows 上启动程序时闪退，应该先检查什么？
```

然后：

1. 左侧先点 **优化**
2. 右侧点 **测试全部**
3. 结果出来后，选一列做 **结果评估**
4. 如果已经跑了两列或更多，再做 **对比评估**

### 例子 B：用户提示词工作区

进入 `/#/basic/user`。

左侧输入：

```text
写一首关于秋天的诗。
```

然后：

1. 左侧先点 **优化**
2. 右侧直接运行测试

这里右侧没有额外测试文本是正常设计，因为被测试的就是这条用户提示词本身。

### 例子 C：文生图工作区

进入 `/#/image/text2image`。

左侧输入：

```text
一只站在雨夜霓虹街头的橘猫，电影感，低饱和，细节丰富。
```

然后：

1. 左侧选择文本模型和图像优化模板
2. 左侧点 **优化**
3. 右侧每列选择图像模型
4. 点击 **测试全部**
5. 比较不同版本或不同图像模型的出图差异

!!! note
    图片质量目前仍建议以人工看图和人工比对为主。这份快速开始先聚焦“如何正确生成和比较”，不把图像质量判断包装成纯自动评估。

### 例子 D：多图生图工作区

进入 `/#/image/multiimage`。

先上传至少两张图，例如：

- 图1：人物参考
- 图2：服装或风格参考

左侧输入：

```text
请以图1的人物为主体，保留图2的服装风格和材质质感，生成一张统一、自然、完整的新画面。确保人物身份稳定，不要把两张图机械拼贴。
```

然后：

1. 先拖拽卡片确定 `图1 / 图2` 顺序
2. 如果传错图，用右上角 `X` 删除
3. 左侧点 **优化**
4. 右侧每列选择支持多图输入的图像模型
5. 先固定模型，比 `原始 / 工作区 / vN`
6. 再固定提示词版本，比不同图像模型

## 最常见的 3 个误区

### “Web 版部署”是不是代理层

不是。Web 版只是前端页面，浏览器还是会直接访问模型服务。

### 为什么我连不上本地模型

通常是浏览器跨域或 HTTPS/HTTP 混合内容限制。优先看 [连接问题](../help/connection-issues.md)，大多数情况下桌面版更合适。

### 清浏览器缓存会不会丢数据

会。浏览器版数据默认保存在本地存储里，清站点数据前请先用 [数据管理](../basic/data.md) 导出备份。
