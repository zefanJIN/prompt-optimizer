# 教学培训

教学培训类任务既可能是一次性的内容生成，也可能是长期复用的课程模板，所以常见用法通常分成两类：

- 临时生成某份内容时，用 `basic/user`
- 需要反复替换主题、对象或难度时，用 `pro/variable`

## 范式一：课程大纲

### 原始写法

```text
设计一个培训课程
```

### 更稳的写法

```text
请为“新员工项目管理入门”设计一份 2 小时培训大纲。

要求：
1. 受众是第一次接触项目管理的新人。
2. 输出结构固定为：培训目标、模块安排、每模块时长、练习建议。
3. 内容要偏实操，不要写成理论教材目录。
4. 语言简洁，方便直接用于备课。
```

## 范式二：练习题或测验

```text
请为“提示词工程基础”设计一组练习题。

要求：
1. 包含 5 道单选题和 2 道简答题。
2. 每道题后给出标准答案。
3. 题目难度面向入门者。
4. 不要超出给定知识范围。
```

这类任务很适合右侧做单结果评估，因为你通常能明确判断：

- 是否覆盖了你要的知识点
- 难度是否合适
- 结构是否完整
- 是否出现了多余解释

## 范式三：可复用课程模板

如果你会反复替换课程主题、时长和对象，可以把提示词写成变量模板：

```text
请为{{课程主题}}设计一份{{时长}}的培训方案。

受众：{{受众}}
难度：{{难度}}

输出要求：
1. 培训目标
2. 内容模块
3. 时间分配
4. 互动练习
5. 课后作业
```

### 适合模式

- `pro/variable`

## 范式四：固定教学风格

如果你希望模型长期保持“像老师一样讲解”的风格，可以加系统提示词：

```text
你是一名面向职场新人的培训讲师。
解释时请循序渐进，优先举具体例子，不要用过度学术化语言。
```

再把具体课程请求放在用户提示词里。

## 使用建议

- 教学任务最怕“结构对了但内容不可教”，所以提示词里最好明确受众和使用场景。
- 如果输出是要直接给学员看，必须写清楚是否允许附加解释。
- 如果你要比较不同提示词或模型的教学效果，右侧评估建议重点看“清晰度、层次感、可执行性”。
